# architecture.md — 입력 → 처리 → 출력

> `GA 프로모션ON`의 v1 구조는 월별 3개 운영 파일(프로모션 총괄판, PPT 프로모션 장표 템플릿, 점포현황)을 기준 입력으로 삼고, 업로드된 값을 하나의 홈페이지형 공유 URL에서 전체 사용자가 확인할 수 있게 제공하는 흐름이다.

| 섹션 | 내용 |
|---|---|
| Goal | RM·영업추진P가 월별 3개 파일만 올리면, 카톡에는 파일이 아닌 홈페이지 URL 하나만 배포하고 모든 사용자가 같은 화면에서 월별·대리점별 프로모션과 JPG를 확인한다 |
| Input | ① 프로모션 총괄판: 대리점별 확정 수치와 협의 상태 ② PPT 프로모션 장표 템플릿: 최종 장표 배경·양식 ③ 점포현황: 지역단·지점·대리점·조직번호 매칭 |
| Process | ① 월 선택 → ② 3개 파일 업로드 → ③ 필수 컬럼·파일 형식 검증 → ④ Supabase 월별 저장 → ⑤ 점포현황 기준 지점/대리점 매칭 → ⑥ 총괄판 수치로 협의중/확정 상태 판정 → ⑦ PPT 템플릿 기준 최종 미리보기/JPG·PPT 생성 |
| Tools | 엑셀 파서, PPT 템플릿 저장소, 검증 스키마, Supabase 데이터 저장소, 홈페이지형 대시보드, 검색/필터, JPG/PPT 렌더러, 관리자 업로드 모드 |
| Output | 단일 홈페이지형 공유 URL + 프로모션 목록 대시보드 + 업로드 검증 결과 + 대리점별 확정 포스터형 화면 + 최종 JPG 다운로드 |
| HITL | 메인 유저의 업로드 결과 검토, 협의 완료 수치 승인, 홈페이지 노출 승인, 최종 JPG 배포 승인. 일반 사용자는 승인권한 없이 확정본 조회·다운로드만 수행 |
| Failure | 엑셀 필수 컬럼 누락 → 업로드 실패 사유 표시 / 협의중 데이터 → 숫자 비공개·JPG 생성 제한 / 승인 전 데이터 → 홈페이지 노출 제한 / JPG 생성 실패 → 화면 미리보기와 재생성 요청 제공 |
| Cost | PoC는 로컬 더미 데이터·정적 화면으로 검증하고, 실제 운영 전 인증·저장소·렌더링 비용을 별도 검토 |

## 데이터 흐름
```mermaid
flowchart LR
  A["월별 3개 파일 업로드"] --> B["총괄판/PPT 템플릿/점포현황 검증"]
  B --> C["Supabase 월별 저장"]
  C --> D["점포현황 기준 지점·대리점 매칭"]
  D --> E["총괄판 수치로 협의중/확정 판정"]
  E --> F["공유 홈페이지 목록 갱신"]
  F --> G["PPT 템플릿 기준 최종 장표 생성"]
  G --> H["대리점별 JPG/PPT 다운로드"]
```
